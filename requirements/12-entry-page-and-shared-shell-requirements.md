# 入口页与公共壳层重构需求文档

## 1. 背景

当前项目已经形成三条顶级玩法线：

- 练习模式
- 闯关模式
- 生存模式

但页面架构仍保留早期形态：

1. `index.html` 既承担网站入口，又直接承载练习模式页面结构与流程
2. `app.js` 既承担练习模式业务逻辑，又默认依附于首页入口
3. `index.html`、`stage-mode.html`、`survival-mode.html` 都各自内嵌了一份高度相似的侧边栏与设置弹窗
4. 三个脚本分别维护设置弹窗的打开、关闭、级别渲染、焦点恢复和禁用逻辑

这会带来两个明显问题：

- 首页职责混乱：入口层与具体玩法层耦合
- 公共壳层重复：后续新增菜单项或调整设置交互时，需要同步修改多份页面与脚本

因此，本次需求希望完成一次明确的页面职责梳理，并为后续继续扩展玩法与内容页打好结构基础。

## 2. 目标

本次需求需要达成以下目标：

1. 让 `index.html` 回归网站入口页，只负责玩法介绍与寻页导航
2. 把练习模式从 `index.html` 中拆出，迁移为独立页面
3. 把侧边栏、设置弹窗与公共级别切换逻辑抽为共享壳层
4. 让顶级页面之间的导航结构统一，避免重复维护
5. 让后续新增菜单项时，尽量只改一处共享配置，而不影响现有玩法业务
6. 在不改变练习、闯关、生存核心玩法规则的前提下完成结构升级

## 3. 范围

### 3.1 本次范围内

- `index.html` 调整为纯入口页
- 新增独立练习模式页面（建议 `practice-mode.html`）
- 提炼公共侧边栏与公共设置弹窗
- 提炼公共级别切换交互
- 统一顶级页面的导航渲染方式
- 明确公共壳层与玩法脚本的职责边界

### 3.2 本次范围外

- 不改练习模式的词义选择与听写规则
- 不改闯关模式的关卡与解锁规则
- 不改生存模式的不限题量与答错即结束规则
- 不引入前端框架、模板引擎或打包构建系统
- 不引入后端路由、账号系统或云端同步
- 不重做现有统计与存储结构，只处理与公共壳层直接相关的部分

## 4. 核心需求

### 4.1 首页职责回归入口层

`index.html` 重构后应满足以下要求：

1. 只作为网站入口页使用
2. 负责展示项目简介、玩法说明与跳转入口
3. 不再直接承载练习模式的首页 / 游戏 / 结果三段式业务结构
4. 不再直接耦合练习模式题目流程
5. 用户可以从入口页清晰跳转到练习模式、闯关模式和生存模式

约束：

- `index.html` 不应继续作为练习模式实际运行页
- 入口页可展示玩法卡片、简介文案、当前级别说明，但不应包含实际出题与结算流程

### 4.2 练习模式独立页面

练习模式应从入口页中拆出，形成单独页面。

要求：

1. 练习模式拥有独立页面入口，建议命名为 `practice-mode.html`
2. 词义选择与听写仍由练习模式页面承载
3. 练习模式现有统计、错词与玩法逻辑继续保留
4. 用户可从入口页与公共侧边栏进入练习模式

说明：

- 首次重构时可暂时保留 `app.js` 文件名以降低迁移风险
- 若后续需要提升命名一致性，可再把 `app.js` 重命名为 `practice-mode.js`

### 4.3 公共侧边栏与设置弹窗

当前各玩法页中重复出现的以下内容，应抽为共享壳层：

- 左侧品牌区
- 左侧主导航
- 当前级别展示区
- 设置弹窗
- 级别选项渲染区

要求：

1. 顶级玩法页不再各自维护一份完整重复的侧边栏 HTML
2. 顶级玩法页不再各自维护一份完整重复的设置弹窗 HTML
3. 公共壳层应支持根据当前页面高亮对应菜单项
4. 公共壳层应支持复用设置弹窗与级别选择能力

### 4.4 公共壳层行为要求

除了结构复用，以下交互行为也需要统一：

1. 打开设置弹窗
2. 关闭设置弹窗
3. 渲染级别列表
4. 保存级别
5. 取消级别修改
6. 弹窗焦点恢复
7. 游戏进行中禁用设置按钮

要求：

1. 上述公共行为不应继续分别散落在 `app.js`、`stage-mode.js`、`survival-mode.js` 中重复实现
2. 玩法脚本只保留和当前玩法直接相关的状态与刷新逻辑
3. 公共壳层负责通用 UI 交互；玩法脚本负责收到级别变化后的业务刷新

### 4.5 导航改为配置驱动

为了让后续新增菜单项不影响现有业务，导航应改为配置驱动。

要求：

1. 菜单项定义集中在共享层统一维护
2. 页面高亮态根据当前页面标识自动生成
3. 新增一个顶级菜单项时，不应要求同时手改所有业务页面中的重复导航 HTML
4. 菜单项文案、链接地址与排序应从共享配置生成
5. 各玩法页必须提供显式返回入口页的导航项，不能只依赖品牌标题跳转

### 4.6 页面职责边界

重构后需要形成清晰边界：

- `index.html`：入口与导航
- 共享壳层：公共导航、设置弹窗、公共级别切换
- `practice-mode.html`：练习模式业务
- `stage-mode.html`：闯关模式业务
- `survival-mode.html`：生存模式业务

要求：

1. 入口页不再拥有练习模式特有状态
2. 共享壳层不直接接管各玩法的题目循环、结算与统计逻辑
3. 各玩法脚本不再负责维护完整重复的公共菜单与设置弹窗结构

### 4.7 全局级别设置继续共享

本次架构重构不改变原有“级别全局共享”的产品约束。

要求：

1. 用户修改级别后，练习模式、闯关模式、生存模式继续共享该级别
2. 页面切换不应丢失级别设置
3. 公共壳层需要成为级别设置的统一入口
4. 各玩法页面在接收到级别变化后，刷新自己的首页、统计或题库视图

### 4.8 静态项目兼容约束

当前项目是纯 HTML / CSS / JavaScript 静态页面项目，本次重构必须兼容这一现实。

要求：

1. 不依赖前端框架
2. 不依赖构建流程
3. 不依赖服务端模板拼接
4. 共享壳层方案必须能在静态页面中直接运行

## 5. 用户故事

### 用户故事 1

作为一个第一次进入网站的用户，我希望 `index.html` 是清晰的入口页，而不是直接跳进某一种玩法，这样我可以先理解有哪些玩法可选。

### 用户故事 2

作为一个想直接练单词的用户，我希望练习模式有自己的独立页面，这样入口结构和玩法结构不会混在一起。

### 用户故事 3

作为一个维护项目的人，我希望侧边栏和设置弹窗只维护一份，这样后续新增菜单项或改设置交互时，不需要同步修改多页代码。

### 用户故事 4

作为一个继续扩展玩法的开发者，我希望导航是配置驱动的，这样未来新增新模式或新入口时，不会牵扯已有玩法逻辑。

### 用户故事 5

作为一个正在进行游戏的用户，我希望在局内禁用设置按钮，以避免中途误改级别造成当前局状态混乱。

## 6. 关键问题澄清

### 6.1 “首页只做入口”含义

“首页只做入口”指：

- 首页负责介绍与跳转
- 首页不再直接运行练习玩法
- 首页不是练习模式别名

### 6.2 “公共组件”在当前项目中的含义

由于当前项目没有框架和模板引擎，这里的“公共组件”不要求引入 React、Vue 或 Web Components。  
本次需求中的“公共组件”，本质上是指：

- 一套共享模板
- 一套共享配置
- 一套共享交互逻辑
- 能被多个静态页面复用的公共壳层

### 6.3 “新增菜单项不影响现有业务”含义

这并不表示以后新增页面时零改动，而是指：

- 不再需要逐页复制和同步修改菜单 HTML
- 不再需要在每个玩法脚本里重复接一套设置弹窗逻辑
- 新增菜单项主要改共享配置与目标页面接入即可

## 7. 验收标准

当满足以下条件时，本次需求分析可视为完成并具备实施基础：

1. 已明确 `index.html` 回归纯入口页
2. 已明确练习模式迁移为独立页面
3. 已明确侧边栏与设置弹窗需要抽为共享壳层
4. 已明确公共设置行为需要统一收口
5. 已明确导航应改为配置驱动
6. 已明确公共壳层与各玩法脚本的职责边界
7. 已明确全局级别设置继续共享
8. 已明确方案必须兼容当前纯静态项目形态
