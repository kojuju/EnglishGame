# 本地用户档案系统需求文档

## 1. 背景

当前项目已经具备以下能力：

- 练习模式：词义选择、听写、错题再战
- 闯关模式：按级别保存解锁进度与每关成绩
- 生存模式：按级别保存最佳纪录与错词记录
- 公共壳层：统一侧边栏、设置弹窗与级别切换
- 本地存储：通过浏览器 `localStorage` 持久化设置、统计、进度与错词

但现阶段所有持久化数据默认都属于“当前浏览器唯一使用者”，会带来以下问题：

1. 同一台设备多人使用时会串档
2. 同一人不同学习用途无法分档保存
3. 当前“全局级别共享”实际上是浏览器全局共享，而不是按用户归属
4. 后续如果要扩展真实账号系统，缺少“数据属于谁”的本地结构基础

因此，本次需求希望引入“本地用户档案”能力，在不引入后端和登录系统的前提下，让现有学习数据按档案隔离保存。

## 2. 目标

本次需求需要达成以下目标：

1. 在同一浏览器内支持多个本地学习档案
2. 支持创建、切换、删除、重命名档案
3. 让每个档案拥有独立的级别设置、练习统计、错词、闯关进度和生存纪录
4. 保持当前练习、闯关、生存核心玩法规则不变，只改变数据归属维度
5. 明确长期学习数据与局内瞬时状态的边界
6. 保持项目仍为纯静态 HTML / CSS / JavaScript 方案，不引入后端服务

## 3. 范围

### 3.1 本次范围内

- 本地档案创建
- 本地档案切换
- 本地档案删除
- 本地档案重命名
- 当只剩最后一个档案时，以“重置档案”替代“删除档案”
- 按档案隔离保存当前项目已有的长期学习数据
- 在切换档案时处理局内状态冲突
- 明确错题再战只属于当前档案当前局上下文
- 首版继续使用浏览器 `localStorage` 作为档案持久化载体
- 在共享壳层中提供统一的档案管理入口与弹窗交互

### 3.2 本次范围外

- 登录 / 注册 / 密码
- 云端同步
- 跨设备同步
- 排行榜、好友系统、分享战绩
- 恢复未完成对局
- 持久化保存局内题号、剩余时间、当前题选项顺序等瞬时状态
- 历史错题本生命周期管理
- 把错题再战升级为跨会话、跨档案的长期复习体系
- 把档案数据静默保存到项目目录下的独立文件夹或 JSON 文件
- 实现不依赖浏览器存储权限的本地文件系统读写

## 4. 核心定义

### 4.1 本地用户档案

“本地用户档案”指：

- 仅存在于当前浏览器本地存储中的用户数据容器
- 每个档案拥有自己的展示名称和独立学习记录
- 不同档案之间数据相互隔离
- 不具备账号身份认证含义

### 4.2 档案内全局共享

本次需求中的“全局共享”需要重新定义为：

- 同一档案内，练习模式、闯关模式和生存模式继续共享当前级别设置
- 不同档案之间的级别设置彼此独立

### 4.3 长期学习数据

长期学习数据指需要跨页面、跨会话保留的数据，例如：

- 当前级别
- 当前练习玩法
- 练习统计
- 错词记录
- 闯关进度
- 生存纪录

### 4.4 局内瞬时状态

局内瞬时状态指仅在当前一局过程中有效的数据，例如：

- 当前题号
- 当前题对象
- 当前选项顺序
- 当前倒计时剩余秒数
- 当前局临时得分、连击、生命
- 当前局未结算的临时错词集合

上述瞬时状态不属于本次档案持久化范围。

## 5. 核心需求

### 5.1 档案基础能力

系统需要支持以下基础能力：

- 创建档案
- 切换档案
- 重命名档案
- 删除档案

要求：

1. 档案名称允许被修改
2. 档案真实身份不能依赖名称，而应依赖内部稳定标识
3. 当前页面需要能够明确展示“当前档案”

### 5.2 档案管理入口与承载形式

首版档案管理不单独新增一个完整管理页面，而是复用现有共享壳层的弹窗能力。

要求：

1. 档案管理入口统一放在共享壳层中
2. 首页、练习模式、闯关模式、生存模式都可进入同一套档案管理 UI
3. 壳层中需要明确展示“当前档案”
4. 首版档案管理以弹窗承载，不要求新增独立路由或独立页面

### 5.3 档案管理弹窗交互

档案管理弹窗需要至少包含以下区域：

1. 当前档案摘要区
2. 档案列表区
3. 选中档案操作区
4. 新建档案输入区

交互要求：

1. 档案列表项需要支持“当前档案”标记
2. 档案列表项需要支持“选中态”
3. 新建档案通过输入名称并点击创建完成
4. 重命名通过编辑态完成，并提供保存 / 取消
5. 删除和重置属于危险操作，需要显式确认
6. 当只剩最后一个档案时，弹窗中不展示“删除档案”，而展示“重置档案”

### 5.4 首次进入与默认档案

系统首次进入时，需要保证用户始终处于某个可用档案之下。

要求：

1. 如果本地尚无档案，系统需要创建首个可用档案，或引导用户创建首个档案
2. 任何时刻系统都必须存在一个“当前档案”
3. 已存在旧版单用户本地数据时，上线档案系统后不应直接丢失原有学习记录

### 5.5 创建档案

用户需要能够新建新的本地档案。

要求：

1. 用户可输入档案名称
2. 创建成功后，系统新增一个独立档案容器
3. 新档案初始为干净学习数据
4. 新档案拥有默认设置值，而不是继承其他档案的统计和进度
5. 创建成功后不要求自动切换到新档案，是否切换由用户单独触发

### 5.6 切换档案

用户需要能够在多个档案之间切换。

要求：

1. 切换后，页面必须刷新为目标档案的数据视图
2. 练习统计、错词、闯关进度、生存纪录都必须同步切换
3. 如果当前不在局内，可直接切换
4. 如果当前正在局内，不允许直接切换，必须先退出或确认放弃本局
5. 放弃本局后，不要求保存当前局的瞬时状态
6. 删除当前档案如果会导致隐式切换到其他档案，也应遵守同样的局内保护规则

### 5.7 重命名档案

用户需要能够修改档案展示名称。

要求：

1. 重命名只影响展示名称
2. 重命名不改变档案内部数据
3. 重命名不创建新档案，也不重置现有学习记录

### 5.8 删除档案

用户需要能够删除不再需要的档案。

要求：

1. 删除档案需要二次确认
2. 删除后，该档案下的长期学习数据一并移除
3. 系统至少保留一个档案
4. 当本地存在多个档案时，允许删除其中任意一个档案
5. 删除当前档案时，需要切换到另一个现存档案

### 5.9 最后一个档案的处理

系统不能进入“无档案”状态。

要求：

1. 当只剩最后一个档案时，不提供“删除档案”
2. 当只剩最后一个档案时，提供“重置档案”能力
3. “重置档案”保留档案本身，但清空该档案下的学习数据并恢复默认设置

### 5.10 档案内共享设置

当前项目中的共享设置需要下沉到档案维度。

要求：

1. 每个档案拥有独立的当前级别
2. 每个档案拥有独立的当前练习玩法选择
3. 同一档案内，练习、闯关、生存继续共享当前级别
4. 切换档案后，应恢复目标档案自己的设置

### 5.11 档案需要记录的数据

系统至少需要按档案维度记录以下长期数据：

- 档案基础信息
  - 档案标识
  - 档案名称
  - 创建时间
  - 最近使用时间
- 档案设置
  - 当前级别
  - 当前练习玩法
- 练习模式数据
  - 按“玩法 + 级别”保存的统计
  - 按“玩法 + 级别”保存的错词记录
- 闯关模式数据
  - 按“玩法 + 级别”保存的解锁进度
  - 按“玩法 + 级别”保存的每关成绩
- 生存模式数据
  - 按“玩法 + 级别”保存的统计
  - 按“玩法 + 级别”保存的错词记录

### 5.12 不记录的瞬时数据

系统不应把以下数据持久化到档案中：

- 当前局题目数组
- 当前题序号
- 当前题选项状态
- 当前倒计时
- 当前连击、生命、临时得分
- 当前局尚未结算的临时错题集合

原因：

- 本次档案系统的目标是隔离长期学习数据，而不是支持断点续玩

### 5.13 错题再战边界

错题再战属于当前档案当前局的临时复习上下文。

要求：

1. 错题再战不能跨档案保留
2. 切换档案后，不再保留上一档案当前局的错题再战上下文
3. 错题再战不应写入档案长期存储结构
4. 刷新页面后，不要求恢复错题再战上下文

### 5.14 首版存储策略

首版档案系统采用浏览器 `localStorage` 作为持久化载体。

要求：

1. 档案数据统一保存到共享层约定的根键中，而不是继续散落为多组浏览器全局键
2. 练习、闯关、生存页面不应直接拼写新的档案存储路径，而应通过共享存储 API 访问
3. 首版不要求把档案保存为项目目录下的独立 JSON 文件
4. 后续若要升级为文件存储或本地服务存储，当前版本应尽量保留统一存储抽象层

## 6. 用户故事

### 用户故事 1

作为一个和家人共用同一台设备的用户，我希望每个人都能有自己的本地档案，这样练习统计、闯关进度和生存纪录不会互相覆盖。

### 用户故事 2

作为一个同时有不同学习目的的用户，我希望能够建立多个档案，例如“日常练习”和“考试冲刺”，这样不同记录不会混在一起。

### 用户故事 3

作为一个正在游戏中的用户，我希望切换档案时系统先要求我退出或确认放弃本局，这样不会让我误以为本局还能被保留。

### 用户故事 4

作为一个长期使用系统的用户，我希望就算只剩最后一个档案，系统也不要让我把所有入口删空，而是提供“重置档案”来重新开始。

### 用户故事 5

作为一个刚刚结束一局、准备做错题再战的用户，我希望这个复习入口只属于当前档案当前局，而不会因为切换到别的档案而混进去。

## 7. 关键问题澄清

### 7.1 本地档案不是登录账号

本次“本地用户档案”不等同于真实账号系统。

本次明确不包含：

- 认证身份
- 密码
- 云端同步
- 跨设备恢复

### 7.2 切换档案为什么不保留本局

本次需求已经明确：

- 当前档案不记录本局瞬时状态
- 切换档案前必须退出或确认放弃本局

因此：

- 切换后不恢复上一档案的未完成对局
- 本次不支持断点续玩

### 7.3 为什么最后一个档案不能删除

因为系统需要始终保持至少一个有效档案作为当前数据归属容器。

所以：

- 多档案状态下可以删除
- 只剩最后一个档案时只能重置，不能删除

### 7.4 错题再战和历史错词的区别

本次“错题再战”是：

- 当前档案
- 当前这一局
- 当前结果页上下文

它不是：

- 历史错题本
- 长期复习列表
- 跨档案共享复习池

### 7.5 为什么首版继续使用 localStorage

当前项目仍是纯静态网页形态。  
在这一前提下，首版优先选择 `localStorage`，原因是：

- 与现有项目形态兼容
- 改造成本最低
- 可以先完成档案能力本身

因此本次明确：

- 首版不实现项目目录下的独立文件夹 / JSON 文件存储
- 后续若需要更稳定的文件级持久化，可作为下一阶段迭代

## 8. 验收标准

满足以下条件时，本次需求可视为分析完成并具备实施基础：

1. 已明确本地档案支持创建、切换、删除、重命名
2. 已明确档案名称允许被修改
3. 已明确切换档案时，局内必须先退出或确认放弃本局
4. 已明确档案不记录局内瞬时状态
5. 已明确最后一个档案不能删除，只能重置
6. 已明确每个档案独立保存练习、闯关、生存相关长期数据
7. 已明确同一档案内继续共享级别，不同档案之间级别独立
8. 已明确错题再战不能跨档案保留
9. 已明确本次不包含登录、云同步和断点续玩
10. 已明确首版档案管理通过共享壳层弹窗承载
11. 已明确首版档案持久化基于 `localStorage`
