# 本地用户档案系统设计文档

## 1. 设计目标

本次设计遵循以下原则：

1. **数据归属收口**：把现有“浏览器全局唯一数据”改造成“当前档案数据”
2. **玩法逻辑不外溢**：练习、闯关、生存规则保持不变，只调整数据读取和写入入口
3. **共享壳层承载入口**：档案管理入口统一放到共享壳层，而不是散落在各业务页
4. **静态项目友好**：方案必须适配当前无框架、无构建、纯静态页面结构
5. **长期与瞬时分离**：档案只保存长期学习数据，不保存局内瞬时状态
6. **渐进迁移**：尽量保留现有业务脚本对共享存储 API 的调用方式，减少一次性重写范围

## 2. 总体信息架构

## 2.1 页面与脚本职责

建议形成如下分层：

| 文件 | 职责 |
| --- | --- |
| `shared-game.js` | 本地档案仓储、当前档案读写、旧数据迁移、对外暴露统一存储 API |
| `shared-layout.js` | 当前档案展示、档案管理入口、档案管理弹窗、公共交互事件 |
| `app.js` | 练习模式业务逻辑，处理局内档案切换确认与切换后刷新 |
| `stage-mode.js` | 闯关模式业务逻辑，处理局内档案切换确认与切换后刷新 |
| `survival-mode.js` | 生存模式业务逻辑，处理局内档案切换确认与切换后刷新 |
| `index.html` | 展示当前档案摘要与玩法入口 |

## 2.2 设计边界

共享层负责：

- 当前档案展示
- 档案列表渲染
- 新建、重命名、删除、重置的公共 UI
- 派发“请求切换档案”事件

玩法页负责：

- 判断当前是否处于局内
- 决定是否允许切换
- 在需要时弹出“放弃本局”确认
- 退出本局并清理瞬时状态
- 在切换完成后刷新业务页面

这样可以保持现有“共享壳层负责公共 UI、玩法页负责业务状态”的职责边界不变。

## 3. 存储模型设计

## 3.1 首版持久化载体

首版明确采用浏览器 `localStorage` 作为档案仓储的实际持久化载体。

### 这样选的原因

1. 当前项目是纯静态 HTML / CSS / JavaScript 项目
2. 现有存储已经建立在浏览器本地存储之上
3. 首版目标是先完成“档案能力”和“数据归属隔离”，而不是解决跨浏览器或文件级持久化

### 本次明确不做

- 不把档案自动保存到项目目录下的独立文件夹
- 不把档案静默写入本地 JSON 文件
- 不引入本地 Python / Node 服务来接管文件读写

### 对后续迭代的要求

虽然首版实际落在 `localStorage`，但业务脚本仍应通过 `shared-game.js` 暴露的统一存储能力访问数据。  
这样后续若替换成文件存储、本地服务或真实账号系统时，可以尽量不改三种玩法页的业务逻辑。

## 3.2 根存储结构

建议新增统一根键：

```js
englishGame.profileStore
```

其内容建议为：

```js
{
  version: 1,
  currentProfileId: "profile-001",
  profiles: {
    "profile-001": {
      id: "profile-001",
      name: "默认档案",
      createdAt: "2026-04-24T10:00:00.000Z",
      lastActiveAt: "2026-04-24T10:00:00.000Z",
      settings: {
        selectedLevel: "cet4",
        selectedMode: "meaning"
      },
      practice: {
        statsByModeAndLevel: {
          meaning: {},
          dictation: {}
        },
        wrongWordsByModeAndLevel: {
          meaning: {},
          dictation: {}
        }
      },
      stage: {
        progressByModeAndLevel: {
          meaning: {},
          dictation: {}
        },
        statsByModeAndLevel: {
          meaning: {},
          dictation: {}
        }
      },
      survival: {
        statsByModeAndLevel: {
          meaning: {},
          dictation: {}
        },
        wrongWordsByModeAndLevel: {
          meaning: {},
          dictation: {}
        }
      }
    }
  }
}
```

## 3.3 这样设计的原因

1. 档案创建、删除、重命名、切换都可以统一在一个对象里完成
2. 后续如果要做导出 / 导入单个档案，结构更完整
3. 所有现有按玩法、按级别的数据都可以自然下沉到档案内部
4. 有利于未来将 `profileId` 迁移为真实 `userId`

## 3.4 数据归属原则

根存储只负责：

- 当前档案是谁
- 所有档案列表
- 每个档案各自的长期数据

不负责：

- 当前局进行到第几题
- 当前局剩余多少秒
- 当前局题目数组和选项顺序
- 当前局错题再战上下文

## 4. 档案仓储设计

## 4.1 `shared-game.js` 的角色变化

当前 `shared-game.js` 已经承担了统一存储入口的角色。  
本次设计建议继续沿用这一中心化思路，把“当前档案”概念收口到这里，而不是让每个玩法页自己拼接存储路径。

## 4.2 建议新增的档案仓储能力

建议新增以下能力：

### 读取能力

- `getProfileStore()`
- `getCurrentProfileId()`
- `getCurrentProfile()`
- `listProfiles()`

### 写入能力

- `createProfile(name)`
- `renameProfile(profileId, name)`
- `deleteProfile(profileId)`
- `resetProfile(profileId)`
- `switchProfile(profileId)`

### 工具能力

- `createEmptyProfile(name)`
- `isLastRemainingProfile(profileId)`
- `touchProfile(profileId)`
- `migrateLegacyStorageToProfileStore()`

## 4.3 与现有 API 的兼容目标

当前业务脚本已经依赖一组共享读写能力，例如：

- `getStoredLevel()`
- `persistSelectedLevel(levelId)`
- `getStoredMode()`
- `persistSelectedMode(modeId)`
- `getModeStats(mode, levelId)`
- `getSavedWrongWords(mode, levelId)`
- `getStageProgress(mode, levelId, totalStages)`
- `saveStageProgress(mode, levelId, progress)`
- `getSurvivalStats(mode, levelId)`

本次设计建议：

- 这些函数的对外语义保持不变
- 内部实现从“全局 localStorage”改为“当前档案仓储”

这样可以显著减少 `app.js`、`stage-mode.js`、`survival-mode.js` 的改动面。

## 5. UI 与交互设计

## 5.1 档案入口位置

建议在公共壳层中新增“当前档案”展示区，与当前级别展示区域并列或相邻。

建议最少展示：

- 当前档案名称
- 一个“管理档案”按钮

这样：

1. 用户在任意页面都能知道当前数据归属
2. 档案管理入口不用在多个页面重复维护

## 5.2 档案管理承载形式

首版不单独新增“档案管理页面”，而是采用“共享壳层入口 + 档案管理弹窗”的方式。

### 这样设计的原因

1. 当前项目已经有公共弹窗交互模式
2. 档案管理属于跨页面公共能力，适合继续放在共享壳层
3. 首版范围聚焦于能力落地，不需要再引入新的路由层

## 5.3 档案管理弹窗结构

建议新增独立的档案管理弹窗，至少包含以下区域：

1. 当前档案摘要
2. 档案列表
3. 新建档案入口
4. 选中档案操作区
   - 切换
   - 重命名
   - 删除
   - 重置（仅最后一个档案时显示）

### 当前档案摘要建议展示

- 当前档案名称
- 最近使用时间
- 当前级别

首版不要求展示复杂统计摘要。

## 5.4 档案列表项展示建议

每个档案项建议展示：

- 档案名称
- 当前档案标记
- 最近使用时间
- 选中态视觉反馈

首版不强制展示：

- 当前级别
- 已通关关卡数
- 生存最佳纪录

这些摘要信息可留待后续扩展。

## 5.5 操作区交互设计

### 新建档案

- 在弹窗内提供名称输入框和“创建档案”按钮
- 创建成功后刷新列表
- 首版不要求创建成功后自动切换

### 切换档案

- 用户先选中一个非当前档案
- 再点击“切换到此档案”
- 这样可以避免列表项点击就直接触发危险切换

### 重命名档案

- 选中档案后点击“重命名”
- 当前项进入编辑态
- 编辑态提供“保存 / 取消”

### 删除档案

- 选中档案后点击“删除档案”
- 弹出二次确认
- 删除按钮使用危险态样式

### 重置档案

- 仅当本地只剩最后一个档案时显示
- 不与“删除档案”同时出现
- 使用危险态样式并要求二次确认

## 5.6 局内保护交互

任何会导致“当前档案切换”的动作，都必须经过局内保护判断。

这至少包括：

- 切换到其他档案
- 删除当前档案后自动切到其他档案

交互建议：

1. 若当前不在局内，动作直接执行
2. 若当前在局内，弹确认“切换将放弃当前局”
3. 用户确认后，先退出本局，再执行目标动作

## 6. 档案切换流程设计

## 6.1 设计约束

共享壳层无法判断各玩法页何时属于“局内状态”，因此不能由壳层直接无条件切换档案。  
真正是否允许切换，需要由当前页面业务脚本判断。

## 6.2 事件流设计

建议采用“两段式事件流”：

### 事件 1：请求切换档案

由共享壳层发起：

```js
window.dispatchEvent(new CustomEvent("english-game:profile-switch-request", {
  detail: { profileId: "profile-002" }
}));
```

### 事件 2：档案切换完成

由共享数据层在切换成功后发出：

```js
window.dispatchEvent(new CustomEvent("english-game:profile-change", {
  detail: { profileId: "profile-002" }
}));
```

## 6.3 切换流程

完整流程建议为：

1. 用户在档案弹窗中点击目标档案“切换”
2. 共享壳层派发 `english-game:profile-switch-request`
3. 当前页面脚本判断是否处于局内
4. 如果不在局内：
   - 直接执行 `switchProfile(profileId)`
5. 如果在局内：
   - 弹出确认“切换将放弃当前局”
   - 用户确认后，先退出本局并清理瞬时状态
   - 再执行 `switchProfile(profileId)`
6. 数据层切换成功后更新 `currentProfileId`
7. 数据层派发 `english-game:profile-change`
8. 共享壳层和当前页面收到事件后重新读取当前档案并刷新 UI

## 6.4 为什么这样设计

1. 壳层无法感知练习、闯关、生存各自的局内状态
2. 玩法页最清楚如何正确“退出本局”
3. 这样可以避免把业务逻辑重新塞回共享层

## 7. 各档案操作的设计细节

## 7.1 创建档案

### 输入

- `name`

### 输出

- 新建后的档案对象

### 默认内容

- 默认级别
- 默认练习玩法
- 空的练习统计与错词
- 空的闯关进度与统计
- 空的生存统计与错词

### 设计建议

- 创建成功后先留在档案列表中
- 是否切换到新档案，交给用户明确触发“切换”动作

这样可以避免“创建即切换”在局内场景下引入额外确认复杂度。

## 7.2 重命名档案

设计要求：

1. 只修改 `name`
2. 不修改 `id`
3. 不清空统计和进度
4. 当前档案和非当前档案都可重命名

由于档案允许重命名，因此内部真实身份必须使用稳定 `id`。

## 7.3 删除档案

设计要求：

1. 删除前必须确认
2. 删除时移除该档案下全部长期数据
3. 若待删档案不是最后一个档案，则允许删除
4. 若删除的是当前档案，则删除前需要先确定一个替代当前档案
5. 若删除当前档案时页面正处于局内，则先走局内放弃确认，再完成删除和切换

### 替代当前档案选择建议

- 优先使用档案列表中的下一个可用档案
- 若不存在下一个，则使用上一个

## 7.4 重置档案

当只剩最后一个档案时，用“重置档案”替代“删除档案”。

### 重置后保留

- `id`
- `name`
- `createdAt`

### 重置后恢复默认或清空

- `lastActiveAt`
- `settings.selectedLevel`
- `settings.selectedMode`
- `practice`
- `stage`
- `survival`

### 这样设计的原因

1. 用户仍保留一个有效档案入口
2. 可以重新开始，而不需要把整个系统置为空状态

## 8. 与玩法业务的衔接设计

## 8.1 练习模式

切换档案后，练习模式需要刷新：

- 当前级别展示
- 当前玩法展示
- 首页统计
- 最近错词
- 开始按钮文案与玩法说明

切换档案时必须清理：

- 当前局题目数组
- 当前局临时错词
- 错题再战上下文
- 当前对局计时与播放副作用

## 8.2 闯关模式

切换档案后，闯关模式需要刷新：

- 当前级别
- 当前解锁关卡
- 已通关关卡
- 关卡列表状态
- 每关历史成绩

切换档案时必须清理：

- 当前关题目进度
- 倒计时
- 当前关临时状态

## 8.3 生存模式

切换档案后，生存模式需要刷新：

- 当前级别
- 生存最佳纪录
- 最近一局表现
- 最近错词

切换档案时必须清理：

- 当前题队列
- 当前题状态
- 当前局临时结果

## 9. 错题再战设计边界

为了满足“错题再战不能跨档案保留”，设计上应明确：

1. 错题再战只存在于页面运行时内存中
2. 不写入档案仓储
3. 不写入 `profileStore`
4. 页面切换档案时，必须主动清理错题再战上下文
5. 页面刷新或关闭后不要求恢复

## 10. 旧数据迁移设计

## 10.1 迁移触发时机

当系统首次读取档案仓储时：

- 如果 `englishGame.profileStore` 不存在
- 但旧版单用户本地存储存在

则触发一次迁移流程。

## 10.2 迁移流程

1. 创建默认档案，例如“默认档案”
2. 读取旧版全局存储中的长期数据：
   - 当前级别
   - 当前练习玩法
   - 练习统计
   - 练习错词
   - 闯关进度
   - 闯关统计
   - 生存统计
   - 生存错词
3. 将这些数据写入默认档案
4. 设置该档案为当前档案
5. 保存新的 `profileStore`

## 10.3 迁移后的策略

- 后续业务统一只读写 `profileStore`
- 旧键可以暂时保留为迁移来源
- 旧键不再作为正式运行期数据源

## 11. 风险点与实现约束

1. **切换流程分散在三个玩法页**
   - 需要统一“局内切换确认”交互语义

2. **存储改造面较广**
   - 现有所有长期数据读写都需要切换为“当前档案路径”

3. **最后一个档案的交互语义**
   - 删除与重置必须在 UI 上明确区分，避免误操作

4. **错题再战语义必须守住**
   - 它是临时复习上下文，不应被误扩展为持久化历史错题体系

## 12. 推荐实现顺序

1. 在 `shared-game.js` 中引入 `profileStore`
2. 完成旧版单用户数据迁移
3. 保持现有共享存储 API 名称不变，内部切到当前档案读写
4. 在 `shared-layout.js` 中加入“当前档案”和“管理档案”入口
5. 实现档案管理弹窗
6. 增加 `profile-switch-request / profile-change` 事件流
7. 在 `app.js`、`stage-mode.js`、`survival-mode.js` 中接入切换确认与刷新逻辑
8. 补齐最后一个档案的“重置档案”分支

## 13. 设计验收标准

满足以下条件时，本次设计可视为完成并具备实施基础：

1. 已形成完整的档案仓储模型
2. 已明确共享层与玩法页在档案切换中的职责分工
3. 已明确局内切换必须先退出或确认放弃本局
4. 已明确长期数据与瞬时状态的存储边界
5. 已明确最后一个档案使用“重置”替代“删除”
6. 已明确错题再战不进入档案长期存储
7. 已明确旧版单用户本地数据的迁移路径
