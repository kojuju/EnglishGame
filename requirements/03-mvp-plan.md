# English Game - MVP 构建计划

## 1. 推荐技术方向

为了更快交付并方便后续迭代，推荐采用以下技术栈：

- Vite
- React
- TypeScript
- 简单 CSS 或 Tailwind
- 本地 JSON 题库
- 浏览器 localStorage 持久化

这套组合足以支撑一个体验完整、无需后端的单人网页 MVP。

## 2. MVP 功能模块

### A. 静态数据模块

负责：

- 词汇数据加载
- 题目来源管理
- 干扰项生成基础

### B. 游戏引擎模块

负责：

- 开始一局
- 题目顺序控制
- 分数更新
- 连击逻辑
- 生命更新
- 计时更新
- 局终条件判断

### C. UI 模块

负责：

- 首页
- 游戏页
- 结果页
- 可复用的状态展示和选项组件

### D. 持久化模块

负责：

- 保存最高分
- 保存最近一局结果
- 保存错词列表

## 5. 里程碑拆分

### 里程碑 1 - 基础框架

- 初始化网页项目
- 建立页面路由或页面状态切换机制
- 准备基础布局与样式

退出条件：

- 应用可以打开，并能在首页、游戏页、结果页之间切换

### 里程碑 2 - 核心游戏流程

- 加载本地词库
- 生成题目集合
- 完成答题交互
- 计算分数、连击和生命
- 在合法条件下结束一局

退出条件：

- 用户能够从开始一直打到结果页，完成完整一局

### 里程碑 3 - 持久化与回顾

- 存储最高分
- 存储最近正确率
- 存储错词列表
- 在结果页展示错词回顾

退出条件：

- 刷新页面后，关键进度数据不会丢失

### 里程碑 4 - 打磨

- 优化反馈动画
- 优化按钮状态
- 优化移动端布局
- 调整文案与节奏

退出条件：

- 产品整体体验协调、顺畅，并具备重复游玩价值

## 6. V1 验收标准

满足以下条件时，MVP 视为完成：

- 用户可以从首页开始一局
- 用户可以完成 10 道词汇题
- 正确和错误反馈都清晰可理解
- 分数、计时、生命、连击按预期工作
- 本局能在正确条件下结束
- 结果页会展示分数、正确率和错词
- 最高分可以本地持久化
- 在桌面和移动端浏览器尺寸下都能正常体验

## 7. 后续待定问题

这些问题不会阻塞 MVP，但后续应重新讨论：

- 每局准确题量是多少
- 单局精确时长是多少
- 错词是否应在同一局中立即再次出现
- 词性是否始终显示
- 答错后是否展示例句

## 8. 紧接着的实现顺序

实际开发顺序建议如下：

1. 创建项目骨架
2. 准备一小套 CET-4 词汇数据
3. 实现游戏引擎
4. 构建三个主页面
5. 加入本地持久化
6. 打磨视觉反馈

## 9. 建议的首批数据规模

一个较合适的起始规模是：

- 总计 120 个单词
- 70 个 easy
- 40 个 medium
- 10 个 hard

这个规模足以验证玩法，又不会拖慢开发节奏。
