# 闯关模式（第一阶段）设计文档

## 1. 设计目标

本次设计遵循以下原则：

1. **练习与闯关分域**：练习模式继续承载自由练习与错题再战；闯关模式承载固定关卡与正式进度
2. **页面物理分离**：闯关模式使用独立的 `stage-mode.html` 页面，避免把新逻辑继续堆进现有首页
3. **设置全局共享**：水平级别仍由统一设置控制，并通过本地存储在不同页面之间共享
4. **进度结构前置扩展**：即便当前只做词义闯关，数据结构也先按 `meaning / dictation` 双桶设计

## 2. 总体信息架构

## 2.1 页面拆分

建议拆为两个入口页面：

- `index.html`：练习模式主页与现有练习流程
- `stage-mode.html`：闯关模式主页、闯关进行页、闯关结算页

### 这样拆分的原因

1. 现有 `index.html` 已经承担练习首页、游戏页、结果页和设置弹窗
2. 闯关模式新增后，会引入关卡选择、解锁进度、每关结果与重试逻辑
3. 如果继续堆在一个 HTML 里，页面状态分支会明显膨胀
4. 单独页面更利于后续继续扩展“听写闯关”

## 2.2 左侧侧边栏

建议将应用外层改造成“左侧导航 + 右侧内容”布局。

左侧导航固定包含：

- 设置
- 练习模式
- 闯关模式

交互建议：

- 在 `index.html` 中，`练习模式` 为当前高亮
- 在 `stage-mode.html` 中，`闯关模式` 为当前高亮
- `设置` 在两个页面中都调用同一套设置弹窗逻辑

## 3. 关卡模式页面结构

`stage-mode.html` 建议包含三个 screen：

1. **闯关首页**
2. **闯关进行页**
3. **闯关结果页**

### 3.1 闯关首页

闯关首页建议展示：

- 当前级别
- 当前闯关玩法：词义闯关
- 当前解锁到第几关
- 已通关关卡数
- 当前可挑战关卡列表
- 进入当前关 / 重玩已通关关卡按钮

第一阶段可优先展示：

- 当前关卡卡片
- 下一待挑战关卡
- 最近已通关若干关

不要求第一版就实现复杂关卡地图。

### 3.2 闯关进行页

闯关进行页复用现有词义题面的核心元素：

- 当前关卡：如 `第 12 关`
- 当前题目进度：如 `第 4 / 10 题`
- 时间
- 分数
- 生命
- 题目英文单词
- 4 个中文选项
- 答题反馈
- 退出本局

### 3.3 闯关结果页

闯关结果页建议展示：

- 关卡编号
- 是否通关
- 最终得分
- 正确率
- 答对题数
- 失败原因（超时 / 血量耗尽 / 正确率不足）
- 是否解锁下一关
- 当前关最高分
- 操作按钮：`重试本关`、`返回闯关首页`

## 4. 关卡数据设计

## 4.1 词库切分策略

每个级别 1000 词，按 10 词一关切分。

结果：

- 每级 100 关
- 每关固定 10 个词

### 推荐实现

新增独立文件，例如：

- `stage-word-bank.js`

但该文件**不建议复制维护完整词条内容**，而建议只维护“关卡分组关系”，例如：

```js
const STAGE_WORD_BANK = {
  meaning: {
    cet4: [
      ["cet4-001", "cet4-002", "cet4-003", "cet4-004", "cet4-005", "cet4-006", "cet4-007", "cet4-008", "cet4-009", "cet4-010"],
      ["cet4-011", "cet4-012", "cet4-013", "cet4-014", "cet4-015", "cet4-016", "cet4-017", "cet4-018", "cet4-019", "cet4-020"]
    ],
    cet6: []
  },
  dictation: {
    cet4: [],
    cet6: []
  }
};
```

实际题面仍从现有 `WORD_BANK` 解析词条详情。

### 这样做的好处

1. 闯关模式拥有独立文件和独立关卡边界
2. 练习模式无需改动词库文件结构
3. 避免维护两套重复词义、音标、词性数据
4. 后续如果要调整某一关的词列表，也只需要改分组文件

## 4.2 题目构建策略

闯关模式某一关的 10 题，来源于该关固定的 10 个词。

但选项中的干扰项不建议只从本关 10 个词中抽取，而建议：

- 正确答案来自本关词
- 干扰项从当前级别的完整词库中抽取

这样做可以避免：

- 同关内选项池过小
- 用户过快记住“本关就这 10 个意思”
- 重试时干扰不足

## 5. 通关规则设计

## 5.1 基础规则

第一阶段闯关模式只做词义选择，建议默认配置：

```js
const STAGE_CONFIG = {
  meaning: {
    questionsPerStage: 10,
    totalTimeSeconds: 75,
    passAccuracy: 90
  }
};
```

## 5.2 血量问题与推荐方案

用户要求包含“血量范围内完成”。

但在 10 题一关、通关正确率至少 90% 的前提下：

- 只允许最多错 1 题
- 如果仍沿用练习模式的 3 血，那么第 2 次错误后虽然血量还没归零，但实际上已经不可能达标

因此，第一阶段推荐将闯关模式的血量单独配置为：

```js
startingLives: 2
```

对应含义：

- 第 1 次错误：仍可继续
- 第 2 次错误：本关立即失败

这样可以让“血量规则”和“90% 通关规则”保持一致。

## 5.3 通关判定

建议通关判定在以下条件同时满足时成立：

1. 已完成本关全部 10 题
2. 没有超时
3. 没有耗尽生命
4. 正确率 `>= 90%`

以 10 题为例：

- 10 / 10：通关
- 9 / 10：通关
- 8 / 10：失败

## 5.4 失败判定

建议失败条件为任一满足即失败：

1. 时间归零
2. 生命归零
3. 完成 10 题后正确率不足 90%

## 6. 重试干扰策略

闯关失败后，重试当前关时：

1. 本关词集合不变
2. 题目顺序重新打乱
3. 每题的选项顺序重新打乱
4. 干扰项重新抽取
5. 尽量避免和上一轮完全相同的选项组合

这套策略可以直接复用当前练习模式“错题再战”里已有的“重抽干扰项 + 选项签名规避”思路。

## 7. 进度与统计存储设计

## 7.1 存储原则

闯关模式需要与练习模式彻底分开存储。

练习模式保留：

- `statsByModeAndLevel`
- `wrongWordsByModeAndLevel`

闯关模式新增：

- `stageProgressByModeAndLevel`
- `stageStatsByModeAndLevel`

## 7.2 建议 storage key

```text
englishGame.stageProgressByModeAndLevel
englishGame.stageStatsByModeAndLevel
```

## 7.3 进度结构示例

```json
{
  "meaning": {
    "cet4": {
      "unlockedStage": 5,
      "completedStages": [1, 2, 3, 4],
      "lastPlayedStage": 5
    }
  },
  "dictation": {
    "cet4": {
      "unlockedStage": 0,
      "completedStages": [],
      "lastPlayedStage": null
    }
  }
}
```

说明：

- `unlockedStage`：当前已解锁到第几关
- `completedStages`：已通关关卡列表
- `lastPlayedStage`：最近进入的关卡

## 7.4 每关统计结构示例

```json
{
  "meaning": {
    "cet4": {
      "1": { "bestScore": 980, "bestAccuracy": 100, "passed": true, "attempts": 1 },
      "2": { "bestScore": 860, "bestAccuracy": 90, "passed": true, "attempts": 3 },
      "3": { "bestScore": 720, "bestAccuracy": 80, "passed": false, "attempts": 2 }
    }
  },
  "dictation": {
    "cet4": {}
  }
}
```

## 8. 页面与脚本拆分建议

建议新增以下文件：

- `stage-mode.html`
- `stage-mode.js`
- `stage-word-bank.js`

职责建议：

- `stage-mode.html`：闯关模式页面结构
- `stage-mode.js`：闯关逻辑、进度读取、关卡构建、通关判定
- `stage-word-bank.js`：关卡分组数据

现有文件职责保持：

- `index.html`：练习模式
- `app.js`：练习模式逻辑
- `words.js`：基础词条真值

## 9. 与现有系统的兼容策略

### 9.1 设置共享

`selectedLevel` 继续沿用当前 localStorage key。

这样：

- 练习模式改级别后，闯关页面自动读取到相同级别
- 闯关页面改级别后，练习模式也自动同步

### 9.2 练习模式保持不变

本次闯关设计不应改动：

- 练习模式首页逻辑
- 听写练习逻辑
- 错题再战逻辑
- 练习统计存储结构

### 9.3 为听写闯关预留

虽然第一阶段不做听写闯关，但以下结构必须预留：

- `stageProgressByModeAndLevel.dictation`
- `stageStatsByModeAndLevel.dictation`
- `STAGE_WORD_BANK.dictation`

这样第二阶段新增听写闯关时，只是补齐页面和流程，而不是推翻存储结构。

## 10. 建议的实现顺序

1. 重构整体布局，给 `index.html` 加左侧侧边栏
2. 抽出共用设置逻辑，使 `index.html` 与 `stage-mode.html` 都能使用
3. 新增 `stage-mode.html` 页面骨架
4. 新增 `stage-word-bank.js`，完成按级别的 10 词分关
5. 实现闯关首页与关卡进入
6. 实现词义闯关答题流程
7. 实现通关 / 失败 / 重试
8. 实现闯关进度与每关统计持久化

## 11. 本轮分析结论

本需求的真正难点不在“本地是否能存进度”，而在于以下三点已经被明确：

1. 闯关必须使用**固定关卡**，不能继续沿用随机练习局
2. 闯关必须与练习**页面分离、存储分离、统计分离**
3. 未来听写闯关虽然暂不实现，但**进度结构必须先分桶设计**

在这个前提下，第一阶段完全可以稳定落地。
