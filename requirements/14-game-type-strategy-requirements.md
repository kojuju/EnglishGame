# 可插拔题型策略层需求文档

## 1. 背景

当前项目已经完成一轮重要的横向拆分：

- `index.html`：入口页
- `practice-mode.html`：练习模式
- `stage-mode.html`：闯关模式
- `survival-mode.html`：生存模式

这意味着“顶级玩法页”已经具备独立边界。  
在这个基础上，下一步自然会遇到一个纵向架构问题：

- `词义选择`
- `听写`

这两种题型是否可以不再只作为练习模式页里的条件分支，而是成为一层可复用、可插拔的“题型策略层”。

从当前代码看，这个问题已经不是抽象讨论，而是现实的维护压力：

1. `app.js` 已经同时承载练习模式页面状态、词义选择逻辑、听写逻辑和结果页逻辑
2. 词义与听写在 `app.js` 中通过大量 `selectedMode === "dictation"` / `selectedMode === "meaning"` 分叉实现
3. `stage-mode.js` 与 `survival-mode.js` 目前只实现了 `meaning`，但存储结构、配置结构和关卡数据都已经为 `dictation` 预留了桶位
4. `shared-game.js` 里已经存在按 `meaning / dictation` 分桶的存储结构，但题型层能力仍偏向 `meaning`

因此，本次需求的核心不是“要不要继续支持两种题型”，而是：

**是否要把题型从页面脚本内部的条件分支，升级为一层可以被不同玩法页组合调用的独立策略能力。**

## 2. 目标

本次需求分析与设计需要解决以下目标：

1. 明确当前项目是否适合引入“可插拔题型策略层”
2. 明确题型层与顶级玩法页之间的职责边界
3. 识别当前代码中已经具备的基础与尚未抽离的耦合点
4. 为后续在 `practice / stage / survival` 三条玩法线上复用题型能力打下结构基础
5. 避免把“词义 / 听写”继续做成散落在各页面脚本里的条件分支集合
6. 为未来补齐 `stage + dictation`、`survival + dictation` 预留实现路径

## 3. 范围

### 3.1 本次范围内

- 研判现有 `practice / stage / survival` 代码的题型耦合情况
- 定义题型策略层的目标职责
- 定义顶级玩法页与题型策略层的交互边界
- 设计题型注册、选择、调用和扩展方式
- 设计渐进迁移方案

### 3.2 本次范围外

- 本次不直接实现 `stage` 的听写题型
- 本次不直接实现 `survival` 的听写题型
- 本次不要求重写所有现有页面脚本
- 本次不改变练习、闯关、生存各自的产品规则
- 本次不引入框架、构建工具或复杂依赖注入系统

## 4. 当前代码研判

### 4.1 当前练习模式的现状

`practice-mode.html + app.js` 是目前题型层最复杂、也最值得优先收口的地方。

从代码现状看：

1. `app.js` 已经存在 `MODE_OPTIONS`、`MODE_META`、`selectedMode`
2. 词义与听写已经是“概念上的两种题型”
3. 但它们不是以独立策略对象存在，而是分散在多个流程函数中的条件分支

这些分支至少覆盖：

- 出题构建
- 题面渲染
- 状态栏刷新
- 作答处理
- 回合推进
- 结果页渲染
- 统计保存
- 语音播放与倒计时副作用

这说明：

- 当前实现已经具备“题型分层”的语义前提
- 但还没有形成真正的题型边界

### 4.2 当前闯关模式的现状

`stage-mode.js` 目前本质上是一个 **meaning-only** 的玩法实现。

表现为：

1. 关卡来源直接取 `STAGE_WORD_BANK.meaning[level]`
2. 进度、统计与结果逻辑也默认按 `meaning` 运行
3. 页面题面、状态栏和判题完全针对词义选择设计

但与此同时，代码里又已经存在以下预留：

- `STAGE_CONFIG.meaning / dictation`
- `STAGE_WORD_BANK.meaning / dictation`
- 按 `mode + level` 分桶的闯关进度和统计结构

这说明：

- 闯关模式已经有“玩法页 + 题型桶”的数据基础
- 只是题型调用层尚未抽成独立接口

### 4.3 当前生存模式的现状

`survival-mode.js` 当前也是一个 **meaning-only** 的玩法实现。

表现为：

1. `SURVIVAL_MODE` 当前固定为 `meaning`
2. 出题、题面、判题和状态栏均围绕词义选择构建
3. 无限续题逻辑直接依赖 `createMeaningQuestion`

但同时，`shared-game.js` 又已经为生存模式预留了：

- `survivalStatsByModeAndLevel`
- `survivalWrongWordsByModeAndLevel`
- `SURVIVAL_CONFIG.meaning / dictation`

这说明：

- 生存模式的数据结构已面向多题型预留
- 题型执行层还没有独立出来

### 4.4 当前共享层的现状

`shared-game.js` 当前的共享层更偏向“基础工具层”，而不是“题型策略层”。

当前已经具备：

- 词库与级别能力
- 存储读写能力
- 按 `meaning / dictation` 分桶的结构约定
- `createMeaningQuestion(...)`

当前尚未具备：

- 统一的题型注册表
- 统一的题型接口
- 通用的“按题型调用构建 / 渲染 / 判题 / 结算”机制
- 听写和词义的并列策略对象

因此，共享层已经有基础，但还不是题型插件层本身。

## 5. 核心需求

### 5.1 题型必须从条件分支升级为独立策略

当前 `meaning / dictation` 不应继续只以 `if / else` 分支散落在页面脚本内部。

要求：

1. 题型应有独立定义单元
2. 页面脚本应通过“选择题型策略”而不是“到处判断 mode”来驱动流程
3. 新增一个题型时，不应要求把出题、渲染、判题、结果文案分别手工塞进多个页面脚本

### 5.2 顶级玩法页与题型层必须分层

引入题型策略层后，必须形成如下职责边界：

- 顶级玩法页：决定一局怎么玩
- 题型策略层：决定一道题怎么出、怎么渲染、怎么判

顶级玩法页负责：

- 回合规则
- 进度推进
- 解锁或纪录
- 本玩法结果页节奏

题型策略层负责：

- 题目数据构建
- 题面渲染
- 作答输入形式
- 判题规则
- 类型专属状态与副作用
- 题型专属结果字段解释

### 5.3 题型层必须支持跨玩法复用

题型层设计不能只服务练习模式。

要求：

1. 题型策略至少应考虑被 `practice / stage / survival` 三条玩法线复用
2. 即便当前只有 `practice + dictation` 已落地，接口也不能写死在练习模式语义上
3. 未来实现 `stage + dictation` 或 `survival + dictation` 时，不应推翻题型层结构

### 5.4 题型层不应接管顶级玩法规则

题型层不能变成新的“大杂烩”。

要求：

1. 题型层不负责闯关解锁
2. 题型层不负责生存模式的无限续题主循环
3. 题型层不负责练习模式“错题再战”这个产品规则本身
4. 题型层只提供与题型本身直接相关的能力

### 5.5 必须兼容当前纯静态项目形态

要求：

1. 不引入框架
2. 不引入构建工具
3. 不要求运行时模块加载器
4. 题型策略层应能通过当前脚本加载方式接入

### 5.6 必须兼容现有存储结构方向

当前很多统计结构已经按 `meaning / dictation` 分桶。

要求：

1. 新题型层应复用这一方向，而不是推翻它
2. 题型标识应与现有桶键保持一致或兼容
3. 不应打破已有练习模式统计数据的可读取性

### 5.7 迁移必须渐进进行

要求：

1. 不要求一步重写 `app.js`
2. 应优先从练习模式抽出题型策略接口
3. 闯关和生存先接入题型注册与选择能力，再逐步补齐更多题型
4. 每一阶段都应保持现有可玩路径不被破坏

## 6. 用户故事

### 用户故事 1

作为维护者，我希望“词义选择”和“听写”是独立题型策略，而不是散落在多个页面脚本里的分支逻辑，这样后续维护时更容易定位修改点。

### 用户故事 2

作为未来要扩展玩法的开发者，我希望同一种题型能被练习模式、闯关模式和生存模式复用，而不是为每个玩法再写一套同题型实现。

### 用户故事 3

作为实现者，我希望新增一个题型时主要补充策略定义，而不是同时改动题目构建、UI、判题、结果和存储的多处重复代码。

### 用户故事 4

作为产品设计者，我希望练习、闯关、生存的玩法规则继续独立，不会因为抽题型层而被重新揉成一个难以理解的大系统。

## 7. 关键问题澄清

### 7.1 为什么这不是 AOP

“词义选择 / 听写”并不是日志、埋点、鉴权这类横切关注点。  
它们会改变：

- 输入形式
- 题面布局
- 判题规则
- 状态栏语义
- 副作用类型
- 结果统计解释

因此，本次更合适的方向是：

- **可插拔题型策略层**

而不是严格意义上的：

- **AOP 切面层**

### 7.2 为什么现在适合做这件事

原因有三点：

1. 顶级玩法页已经拆开，横向边界已经清晰
2. `meaning / dictation` 在存储结构和配置结构上已经有并列语义
3. 练习模式中的 `if / else` 分支已经足够多，继续堆叠会进一步放大维护成本

### 7.3 为什么不能直接把所有逻辑都丢到共享层

因为共享层如果既管词库、存储、壳层，又直接接管所有玩法和题型逻辑，会迅速失控。  
题型层应是一层独立的策略能力，而不是把所有页面逻辑塞进 `shared-game.js`。

## 8. 验收标准

当满足以下条件时，本次需求分析可视为完成并具备实施基础：

1. 已明确当前项目适合引入可插拔题型策略层
2. 已明确这条设计路线不是 AOP，而是策略/插件层
3. 已明确当前 `app.js` 中题型分支的主要耦合点
4. 已明确 `stage-mode.js` 与 `survival-mode.js` 的 meaning-only 现状与 dictation 预留基础
5. 已明确题型层与顶级玩法页的职责边界
6. 已明确题型层需要跨 `practice / stage / survival` 复用
7. 已明确迁移必须渐进进行，并优先从练习模式开始
8. 已明确方案必须兼容当前纯静态项目形态和现有存储方向
